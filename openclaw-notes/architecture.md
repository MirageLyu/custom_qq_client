# OpenClaw 技术架构深度解析

## 一句话定义

OpenClaw 是一个**开源、可自托管的 AI Agent 网关框架**。它的本质是一个**消息路由中枢**，把来自各种聊天平台的消息统一接入，路由给对应的 AI Agent 处理，再把结果回传给用户。

---

## 核心设计：三层架构

OpenClaw 的整体设计遵循**关注点分离**原则，将系统拆分为三个独立的层：

```
┌─────────────────────────────────────────────────────────┐
│                     用户 / 客户端                        │
│         (Telegram, Discord, QQ, WhatsApp, Web...)       │
└────────────────────────┬────────────────────────────────┘
                         │ 消息收发
                         ▼
┌─────────────────────────────────────────────────────────┐
│              Layer 1 · Channel 层（通道适配）             │
│                                                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ Telegram │ │ Discord  │ │ WhatsApp │ │  Custom  │   │
│  │ Adapter  │ │ Adapter  │ │ Adapter  │ │ Adapter  │   │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘   │
│       └─────────────┼───────────┼─────────────┘         │
│                     │  统一消息格式                       │
└─────────────────────┼───────────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────────┐
│              Layer 2 · Gateway 层（网关核心）             │
│                                                         │
│  ┌──────────────────────────────────────────────────┐   │
│  │  Gateway Daemon (Node.js, 默认端口 18789)         │   │
│  │                                                  │   │
│  │  · 消息路由引擎 ── 决定消息交给哪个 Agent          │   │
│  │  · 会话状态管理 ── 维持上下文连续性               │   │
│  │  · Agent 生命周期 ── 创建/销毁/调度               │   │
│  │  · Web 控制面板 ── 可视化管理界面                 │   │
│  │  · WebSocket 长连接 ── 低延迟双向通信             │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────┬───────────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────────┐
│              Layer 3 · LLM 层（模型接口）                │
│                                                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ OpenAI   │ │ Claude   │ │ Gemini   │ │ DeepSeek │   │
│  │ GPT-4o   │ │ Opus/Son │ │  2.x     │ │   V3     │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │
│  ┌──────────┐ ┌──────────┐                              │
│  │  Qwen    │ │ Ollama   │  ← 本地模型也支持             │
│  │          │ │ (本地)   │                              │
│  └──────────┘ └──────────┘                              │
└─────────────────────────────────────────────────────────┘
```

**三层分离的好处**：
- 新增聊天平台？只改 Channel 层，写一个 Adapter 即可
- 切换 AI 模型？只改 LLM 层，Gateway 无感知
- 调整路由策略？只改 Gateway 层，两端不受影响

## 核心组件详解

### 1. Gateway Daemon（网关守护进程）

这是 OpenClaw 的**心脏**。它是一个持久运行的 Node.js 进程，职责包括：

| 职责 | 说明 |
|------|------|
| 消息路由 | 根据平台标识、用户角色、频道等规则，将消息分发给正确的 Agent |
| 会话管理 | 维护跨平台、跨会话的上下文状态 |
| Agent 调度 | 管理 Agent 的创建、激活、休眠和销毁 |
| WebSocket 连接 | 与各 Channel 保持长连接，实现低延迟双向通信（不是轮询） |
| Web UI | 内置管理面板，用于配置和监控 |

### 2. Channel Adapter（通道适配器）

每个聊天平台对应一个插件化的 Adapter，实现统一的 `ChannelPlugin` 接口：

```
ChannelPlugin 接口协议：
├── id          → 插件唯一标识
├── meta        → 元信息（名称、版本、作者）
├── capabilities → 能力声明（文本/图片/语音/文件...）
├── config      → 配置 schema
└── 可选适配器：
    ├── outbound    → 发送消息的实现
    ├── pairing     → 认证流程（如扫码登录）
    ├── groups      → 群组管理
    ├── gateway     → 额外的 Gateway RPC 方法
    └── agentTools  → Agent 可调用的平台特有能力
```

插件按优先级从四个来源加载：
1. 配置文件显式指定的路径（最高）
2. 工作区开发目录
3. 全局 npm 包
4. 内置核心插件（最低）

### 3. LLM 层（模型接口）

模型无关设计——OpenClaw 不绑定任何特定 AI 模型：
- 云端模型：OpenAI、Claude、Gemini、DeepSeek、Qwen 等 50+
- 本地模型：通过 Ollama 接入任意开源模型
- 混合部署：可同时使用云端 + 本地模型，按场景路由

### 4. Memory 系统（记忆系统）

提供跨会话、跨平台的持久化记忆：

```
记忆架构：
├── 短期记忆 ── 当前会话上下文（自动管理）
├── 长期记忆 ── Soul Memory Skill 管理
│   ├── 优先级标签  [C]=关键 [I]=重要 [N]=普通
│   ├── 向量搜索    支持 CJK 分词
│   ├── 动态分类    自学习类别体系
│   ├── 版本控制    Git 集成，可审计
│   ├── 记忆衰减    基于时间的自动清理
│   └── 自动触发    响应前后自动注入上下文
└── 存储后端 ── SQLite / PostgreSQL / Redis
```

### 5. Skills 系统（技能扩展）

Skills 是 OpenClaw 的**插件生态**，通过 ClawHub 社区市场分发（截至 2026 年已有 5700+ 技能）：

```
技能加载优先级：
1. /skills          ── 当前工作区（开发用）
2. ~/.openclaw/skills ── 本地已安装
3. 内置技能           ── 随 OpenClaw 打包
```

安装方式：`clawhub install @author/skill-name`

## 配置哲学：Markdown-as-Config

OpenClaw 的一个独特设计是用**纯文本 Markdown 文件**来定义 Agent 身份和行为，而不是写代码：

| 配置文件 | 作用 |
|----------|------|
| `SOUL.md` | Agent 的"宪法"——人格、价值观、行为边界，每次推理循环最先加载 |
| `AGENTS.md` | Agent 的能力定义和分工 |
| `USER.md` | 用户偏好和个性化设定 |
| `TOOLS.md` | Agent 可使用的工具声明 |
| `IDENTITY.md` | Agent 身份信息 |
| `MEMORY.md` | 记忆配置和策略 |

**SOUL.md 是核心**：它在每个推理周期最先加载，约束所有 Skills 的行为方式。Skills 增加能力，SOUL.md 控制能力如何被使用。

## 本质总结

```
OpenClaw 的本质 = 消息路由网关 + AI Agent 运行时 + 插件生态

它不是一个 AI 模型，而是一个让你：
  1. 统一接入多个聊天平台        （Channel 层解决）
  2. 灵活调用多个 AI 模型        （LLM 层解决）
  3. 以声明式配置定义 Agent 行为  （Markdown-as-Config）
  4. 通过技能市场扩展能力         （Skills/ClawHub）
  5. 维护跨会话跨平台的记忆       （Memory 系统）
的框架和平台。
```

## 与我们项目的关系

当前项目 `qq-client` 是一个 Rust 项目。如果要接入 OpenClaw：
- **方案 A**：为 OpenClaw 编写一个 QQ Channel Adapter（TypeScript），让 OpenClaw 网关管理 QQ 消息
- **方案 B**：在 Rust 项目中直接调用 OpenClaw 的 REST API，将其作为 AI 后端
- **方案 C**：独立部署 OpenClaw 实例，qq-client 通过 WebSocket/HTTP 与之通信

具体选择取决于我们想让 OpenClaw 在整个系统中扮演什么角色。
